# 项目交接与执行规范

## 1. 项目目标
- 项目：`doubleLLM_logit_comflicts`
- 目标：形成可提交的中国发明专利申请包，主题为“**双大模型逻辑分歧的自动检测与结构化解耦算法**”。
- 核心交付：`权利要求书 + 说明书 + 摘要 + 附图 + 现有技术检索对比 + 提交清单`，并输出 `docx` 版本。

## 2. 目录约定
- 工作主目录：`patent_cn/`
- 里程碑：
  - `M0` 初始化目录与占位符
  - `M1` 现有技术检索 Top20 + Top5 claim chart
  - `M2` 差异化特征池 + A/B/C 路线权利要求草案
  - `M3` 说明书草案（可支撑权利要求）
  - `M4` 附图生成 + 引用一致性检查
  - `M5` 自动排版导出 docx + 校验 + 最终打包

## 3. 回复与评审规范
- 每次重要推进必须同步到：`patent_cn/00_admin/work_progress.md`。
- 每次记录必须包含：
  - 时间戳（北京时间）
  - 本次完成内容
  - 风险点与规避动作
  - 文件评审清单（完整性、术语一致性、可实现性、合规性）
- 输出风格：
  - 不写空泛结论，给出可验证事实、路径与结果。
  - 明确“已完成/未完成/待确认”的边界。
  - 避免复制现有专利原文，全部原创改写。

## 4. 技术写作硬规则
- 权利要求用“获取、解析、对齐、判定、生成、输出”等可执行步骤表达。
- 说明书必须闭环：技术问题 -> 技术手段 -> 技术效果。
- AI相关内容必须工程化：数据结构、阈值、流程、资源调度、停止条件、合规策略。
- 至少两类实施例：
  - 无外部证据，仅结构化解耦并保留多解
  - 有外部证据检索与裁决回填

## 5. 自查清单（提交前必须全绿）
- 摘要长度（含标点）`<= 300`
- 权利要求编号连续且相互引用合法
- 说明书术语定义覆盖独立权利要求关键术语
- 附图标记在说明书中的引用一致
- Top20/Top5 对比与差异化路线 A/B/C 已落地到权利要求

## 6. 自动化脚本入口
- 图生成：`python patent_cn/scripts/build_figures.py`
- 文档导出：`python patent_cn/scripts/build_docx.py`
- 文本检查：`python patent_cn/scripts/lint_patent_text.py`
- 摘要检查：`python patent_cn/scripts/check_abstract_len.py`
- 权利要求支撑映射：`python patent_cn/scripts/claim_support_mapper.py`
